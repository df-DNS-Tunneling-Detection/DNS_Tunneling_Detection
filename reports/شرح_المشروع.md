## 1. المشروع بيتكلم عن إيه؟

### المشكلة

الـ **DNS** هو البروتوكول اللي بيحوّل أسماء المواقع (زي `google.com`) لـ IP addresses. اللي بيحصل إن في **كل** شبكة في الدنيا بيسيبوا الـ DNS مفتوح من غير ما يفتّشوا فيه كتير، عشان الإنترنت نفسه ميشتغلش من غيره.

الهاكرز استغلّوا الموضوع ده وعملوا حاجة اسمها **DNS Tunneling**. الفكرة بسيطة:
- بدل ما الـ DNS query يبعت اسم موقع طبيعي، الهاكر بيـ **encode** بيانات (data) جوّه الـ subdomain.
- مثال: بدل `mail.google.com` يبعت `aXk1bG9yZW0gaXBzdW0.tunnel.evil.com` — الجزء الطويل ده base32 encoded payload.
- السيرفر اللي بيستقبل الـ DNS الخبيث بيـ decode الـ payload ويرد بنفس الطريقة.

**النتيجة:** قناة سرّية للـ command-and-control أو تسريب بيانات (data exfiltration)، شبكة الضحية مش حاسة بحاجة لأن كل اللي شايفاه DNS عادي.

### إيه اللي عملناه

بنينا **detector** (مكتشف) باستخدام Machine Learning بيقرّر لكل DNS flow: ده **عادي** (benign) ولا **tunneling** (malicious). استخدمنا اتنين موديل:
1. **Random Forest**
2. **XGBoost**

النتيجة: **دقّة 99.9%** على بيانات حقيقية.

## 2. عملنا إيه بالظبط — خطوة بخطوة

### الخطوة 1: حدّدنا الـ Pipeline

بنينا 5 سكريبتات بايثون كل واحد له شغل محدّد:

```
الـ raw data
    ↓
[preprocess.py]   ← تحميل وتنظيف البيانات
    ↓
[features.py]     ← حساب الـ features
    ↓
[train.py]        ← تدريب الموديلين
    ↓
[evaluate.py]     ← قياس الأداء + رسم plots
[predict.py]      ← استخدام الموديل المُدرَّب لتصنيف بيانات جديدة
```

### الخطوة 2: الـ Dataset (دي قصّة طويلة، شوف القسم 3)

اخترنا في الآخر **CIRA-CIC-DoHBrw-2020** — وهو dataset حقيقي من جامعة New Brunswick الكندية، فيه:
- DoH (DNS over HTTPS) traffic عادي من المتصفّحات (Chrome, Firefox)
- DoH tunneling traffic مولّد بأدوات حقيقية: `iodine`, `dnscat2`, `dns2tcp`

الـ dataset فيه **270 ألف flow**، كل flow بـ 35 column (عمود) فيه إحصائيات الحزم.

### الخطوة 3: الـ Features

في حالتين:

**حالة 1: لما يكون عندنا الـ query string** (نص الـ DNS request)
بنحسب 11 feature لكل query:

| الـ Feature | فكرته إيه |
|---|---|
| `query_length` | طول الـ query (الـ tunneling أطول لأنه فيه payload) |
| `entropy` | Shannon entropy — البيانات الـ encoded عشوائية فالـ entropy عالي |
| `digit_ratio` | نسبة الأرقام — base32 فيه أرقام كتير |
| `subdomain_length` | طول الـ subdomain |
| `longest_label_length` | الـ DNS بيسمح بـ 63 حرف max في الـ label، والـ tunneling غالباً بيوصل للحد |
| `unique_char_count` | عدد الحروف المختلفة |
| `vowel_consonant_ratio` | نسبة الحروف المتحرّكة للساكنة (الأسماء الطبيعية فيها توازن إنجليزي) |
| `bigram_entropy` | entropy لكل زوج حروف |
| `uppercase_ratio` | نسبة الحروف الكبيرة |
| `label_count` | عدد النقاط في الـ domain |
| `non_alphanum_ratio` | نسبة الحروف اللي مش حروف ولا أرقام |

**حالة 2: الـ dataset عنده flow features جاهزة** (CIC-DoHBrw-2020)
هنا مفيش query string. الـ dataset جايب 33 feature إحصائية من الحزم نفسها:
- `PacketLengthMean`, `PacketLengthVariance`, `PacketLengthMode`...
- `FlowBytesSent`, `FlowBytesReceived`, `FlowSentRate`...
- `PacketTimeMean`, `PacketTimeStandardDeviation`...
- `ResponseTimeTimeMean`...

الكود بتاعنا بيشتغل على الحالتين تلقائياً — بيشوف هل في query column ولا لأ، ويتصرف حسب كل حالة.

### الخطوة 4: تدريب الموديل

عملنا الخطوات دي:
1. **قسّمنا البيانات 80/20** train/test، مع **stratified split** (يعني الـ test يحتوي على نفس نسبة الكلاسات).
2. **5-fold Cross-Validation** على الـ training set عشان نتأكّد إن الموديل مش بيـ overfit.
3. درّبنا **Random Forest** (150 شجرة، عمق 20) و **XGBoost** (300 شجرة، عمق 6).
4. حفظنا الموديلين كـ `.pkl` files في فولدر `models/`:
   - `rf.pkl` — 820 كيلوبايت
   - `xgb.pkl` — 60 كيلوبايت

### الخطوة 5: تقييم الأداء

قسنا 5 مقاييس:
- **Accuracy** (الدقّة العامة)
- **Precision** (لما الموديل يقول tunneling، كام مرة بيكون صح؟)
- **Recall** (من كل الـ tunneling الفعلي، الموديل لقى كام؟)
- **F1** (متوسّط بين Precision و Recall)
- **ROC-AUC** (مقياس عام بيشوف الموديل بيفرّق بين الكلاسات إزاي)

ورسمنا plots:
- Confusion Matrix لكل موديل
- ROC curves للمقارنة
- Feature Importance عشان نعرف أهم features

## 6. أسئلة متوقّعة من الدكتور — والإجابات

**س: ليه استخدمت Random Forest و XGBoost بالذات؟**
- **ج:** اتنينهم tree-based ensemble methods — مش محتاجين feature scaling، بيتعاملوا مع correlated features كويس، وأهم حاجة بيقولوا لنا الـ feature importance. ودي أكتر طريقة شائعة في الـ literature للـ tunneling detection.

**س: الدقة عالية أوي... ده طبيعي؟**
- **ج:** آه — الـ paper الأصلي بتاع CIC-DoHBrw-2020 وصلوا لنفس النتائج تقريباً. الـ features الإحصائية على الحزم بتفرّق بقوة بين tunneling و benign. **بس** ده على الـ tools الموجودة في الـ dataset (iodine, dnscat2, dns2tcp) — هاكر يصمّم أداة جديدة بحذر قد يبوظ نتائج الموديل.

**س: ليه ما عملتش Deep Learning؟**
- **ج:** لأن الـ classical ML كافي ومتفوّق بدرجة كبيرة. والـ Random Forest و XGBoost أسهل في الـ explainability — أقدر أشوف feature importance بسهولة. Deep Learning كان هيوصل لنفس النتيجة بـ training time أطول وصعوبة شرح أكتر.

**س: ليه ما استخدمتش Deep Learning مع الـ query strings؟**
- **ج:** في الـ CIC-DoHBrw-2020، البيانات flow features، مش query strings. لو كان الـ dataset فيه queries، الـ Deep Learning كان يبقى أفضل (character-level CNN/LSTM). للـ flow features، الموديلات الـ tree-based الأفضل.

**س: إزاي بتـ generalize على tools جديدة؟**
- **ج:** ده **القصور الرئيسي** في المشروع. الـ dataset فيه 3 tools بس (iodine, dnscat2, dns2tcp). لو في tool جديد بـ encoding مختلف، الموديل ممكن يفشل. الحل في المستقبل: نختبر الموديل على tools ما اتدرّبش عليها (transfer test).

**س: إيه عيوب الموديل؟**
- **ج:** اربع نقط رئيسية:
  1. مش بيشوف الـ raw query string في الـ DoH — لأن الـ TLS بيشفّره.
  2. مولّش لو الـ attacker عمل shaping لـ flow الإحصائيات عشان تبان طبيعية.
  3. مش بياخد في الاعتبار temporal patterns (rate, periodicity).
  4. متدرّب على dataset من tools محدّدة — مش معروف هيـ generalize على tools جديدة ولّا لأ.

**س: ليه CIC-DoHBrw-2020 مش CIC-Bell-DNS-2021؟**
- **ج:** الـ Bell-DNS-2021 محتوياته malicious-domain classification (مش tunneling). أنا اكتشفت ده بعد ما حمّلت الـ dataset وفتحت الـ CSVs.

**س: إيه الـ Shannon entropy وإزاي بتحسبها؟**
- **ج:** $H(X) = -\sum_i p_i \log_2 p_i$
- بنحسب نسبة كل حرف في الـ query، نضربها في الـ log بتاعها، ونجمع كل ده ونشيل الإشارة السالبة.
- الـ DNS العادي entropy بتاعه قليل لأنه لغة طبيعية (حروف معيّنة بتيجي أكتر). الـ encoded payload بيقترب من الـ uniform distribution فالـ entropy بيكون عالي.

**س: ليه استخدمت 5-fold cross-validation؟**
- **ج:** عشان أتأكّد إن الموديل مش بيـ overfit على split معيّن. الـ 5-fold بتقسم الـ training set لـ 5 أجزاء، تدرّب على 4 وتختبر على 1، وتعيد ده 5 مرات.

**س: ليه stratified split؟**
- **ج:** عشان نفس نسبة الكلاسات تتحافظ في الـ train و الـ test. لو عملت split عشوائي، ممكن كل الـ benign يقعوا في الـ train والـ test كله malicious.

**س: ليه `class_weight="balanced"`؟**
- **ج:** عشان الـ class imbalance — الـ Random Forest كان بيتحيّز للكلاس الأكتر. الـ balanced بيدّي وزن أعلى للكلاس الأقل.

## 7. ملخّص النقاط اللي تحبّ تذاكرها كويس قبل المقابلة

1. **اسم الـ Dataset:** CIRA-CIC-DoHBrw-2020 (من University of New Brunswick).
2. **عدد الـ samples:** 270 ألف flow في الـ dataset الكامل، 10 آلاف في الـ sample بتاعك.
3. **عدد الـ features:** 33 numeric flow feature.
4. **الموديلات:** Random Forest (150 trees, max_depth=20) + XGBoost (300 trees, max_depth=6).
5. **النتائج:** Accuracy 99.9%, F1 99.9%, ROC-AUC 1.000.
6. **أهم Feature:** PacketLengthMode (20.5% importance).
7. **الـ Cross-validation:** 5-fold stratified.
8. **الـ Train/Test Split:** 80/20 stratified.
9. **الأدوات اللي درّبنا عليها:** iodine, dnscat2, dns2tcp (موجودة في الـ dataset).
10. **العيب الرئيسي:** الموديل ممكن يفشل مع tool جديد ما اتدرّبش عليه.
